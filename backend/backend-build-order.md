# Backend Build Order

A sequence for what to hand Claude (VS Code) next, phase by phase. Ordering logic, in short: **prove the connection → build the one shared primitive everything else reuses → put the safety pattern (human-in-the-loop) in place before anything auto-writes → layer features on top → chat/AI layer last, since it's a thin wrapper around everything below it.**

Each item below references its number from `ai-calendar-manager-spec.md`'s section 12 table, so you can cross-check the algorithmic/API/AI classification while building.

## Phase 1 — Prove the calendar connection

The immediate next step — everything after this depends on it actually working.

1. ✅ **Done.** **`GET /api/calendar/events`** — list events from the burner calendar using the service account credentials. No scheduling logic yet, just proves the auth chain works end to end (table #19). Implemented via `lib/googleCalendar.ts` (shared JWT client, full `calendar` scope) + `app/api/calendar/events/route.ts`; verified locally against the real burner calendar.
2. ✅ **Done.** **External → burner sync engine** — using `syncToken` for incremental updates and `extendedProperties` source-ID tagging for dedup, per the design already locked into `architecture-plan.md` section 2a (table #19, #20). Implemented via `lib/calendarSync.ts` + `lib/eventMetadata.ts` + `POST /api/calendar/sync`, backed by two new Supabase tables (`calendar_sync_state`, `synced_events`) for resumable, efficient dedup. Source calendars are configured via `GOOGLE_SOURCE_CALENDAR_IDS` (`calendarList.list()` doesn't work for service accounts — see `architecture-plan.md` section 2a), each tagged with a `sourceLabel`; first backfill per calendar capped 1 year out; Google writes retry with backoff to survive the Calendar API's write burst quota. Verified end-to-end against 5 real calendars — full backfill, idempotent re-run (steady-state incremental, zero changes), and correct `sourceLabel` tagging all confirmed.
   - **Bug found & fixed (2026-07-22):** a batched end-of-page `synced_events` flush meant a mid-page function kill could lose already-successful Google writes, causing the next sync to recreate the same events as duplicates — 121 orphaned duplicates were produced this way in one source calendar. Fixed by flushing each event's `synced_events` row immediately after its own Google write succeeds. A maintenance endpoint, `POST /api/calendar/sync/dedupe`, was added to clean up existing duplicates (and any future ones) — see `backend-api-reference.md`. Verified: cleanup ran to zero duplicates, and a subsequent sync produced `created: 0` across all calendars.
   - **Default priority changed to `critical` (2026-07-22):** synced events defaulted to `medium`; changed to `critical` per the user — synced events are always `flexible: 'false'` (hard to move) and shouldn't need to move anyway. Only affects events synced from here on; already-synced events keep whatever priority they were written with until they're next touched by a real upstream change (the sync engine skips unchanged events via etag, so this isn't retroactive).

## Phase 2 — The shared primitive + the safety pattern

Build these before any feature that writes to the calendar — everything in Phase 3 is "apply this primitive with different constraints," and nothing should auto-write without the review queue existing first.

3. ✅ **Done.** **Free-slot finding / conflict detection** (table #1, #2) — the one piece of logic that tasks, habits, focus time, buffer insertion, and the booking page will all reuse. Worth over-investing in correctness here specifically. Implemented as a clean pure/IO split: `lib/intervals.ts` (pure interval math — merge/subtract/pad/filter, unit-tested), `lib/workingHours.ts` (DST-correct working-window generation via `luxon`, unit-tested including real spring-forward/fall-back transitions), `lib/busyIntervals.ts` (Google fetch + per-event-timezone normalization, unit-tested), `lib/freeSlots.ts` (`findFreeSlots`/`detectConflicts` orchestration), `lib/schedulingConfig.ts` (`HOME_TIMEZONE`/`WORKING_HOURS_START`/`WORKING_HOURS_END`/`WORKING_DAYS` env vars, defaults `America/New_York` 10:00–18:00 Mon–Fri). Exposed via `GET /api/calendar/free-slots` and `GET /api/calendar/conflicts` for verification against the real burner calendar. 47 unit tests (`npm test`), every non-cancelled burner event counts as busy regardless of `flexible` tag — deciding what to do about conflicts is explicitly deferred to item 5.
4. ✅ **Done.** **Proposed-changes / review queue** — a data model (separate from calendar events) plus approve/reject endpoints. This is the human-in-the-loop mechanism from the spec's AI Assistant section — every scheduling feature from here on proposes into this queue rather than writing directly. Implemented via a new `proposed_changes` Supabase table + `lib/proposedChanges.ts` + `GET/POST /api/proposed-changes` + `POST /api/proposed-changes/{id}/approve|reject`. Per-`change_type` field validation (`create`/`move`/`update`/`delete`); a 4-state status machine (`pending → applied | rejected | failed`, no separate "approved" state since applying happens synchronously); an `AUTO_APPLY_CATEGORIES` whitelist env var (empty/unset = nothing auto-applies, per your explicit choice to build this now rather than defer it) that applies a proposal immediately at creation time instead of waiting for approval; a conflict recheck via `detectConflicts` (item 3's primitive) on every `create`/`move` apply, failing safe with a descriptive error rather than double-booking. Verified end-to-end against the real burner calendar: manual approve, auto-apply (both success and a real-conflict failure), reject, retry-from-failed, and 409s on already-decided rows.
   - **Data-model alignment + additions (2026-07-22):** clarified the real model with the user — a `tasks` database (not yet built, Phase 3) and calendar events are separate; a block *can* optionally tie back to a task but doesn't have to; synced external events are always non-flexible (already true). Changed `priority` from a numeric `1-5` scale to `critical`/`high`/`medium`/`low` (matches how the user actually thinks about events — doctors' appointments/classes are `critical`); `colorTag` is now always derived from `category` (`lib/eventMetadata.ts`'s `CATEGORY_COLORS`) rather than freely settable, so it's never blank; added a `deadline` field (a "must be done by" constraint, independent of where a block is actually scheduled — no logic reads it yet, that's Phase 3 item 7); `GET /api/calendar/events` now surfaces `category`/`priority`/`deadline`/`colorTag`/`origin` per event; every create/approve/reject response now includes a plain-language `message` alongside `status`. "All-day note vs. block" isn't a separately stored type — it's inferred from Google's own all-day-vs-timed event shape, same as `lib/busyIntervals.ts` already does for scheduling.
5. ✅ **Done.** **Auto-reschedule on conflict** (table #3) — mechanical once #3 and #4 exist: detect overlap, propose a move, wait for approval. Implemented via `lib/autoReschedule.ts` (`findAndProposeReschedules`) + `POST /api/calendar/reschedule`. Pairwise conflict scan over a time window using `intervalsOverlap` (item 3); movability comes from an event's own `flexible` metadata, not `category` — if both sides of a conflict are flexible, the lower-priority one moves (`critical` > `high` > `medium` > `low`, the priority scale from item 4's rework), tie-broken deterministically by `eventId`; unresolvable (both non-flexible) conflicts are reported, not forced. Replacement slot is the first opening `findFreeSlots` returns of at least the event's own duration; proposes via the normal `createProposedChange` (item 4) so `AUTO_APPLY_CATEGORIES` still applies immediately where whitelisted. Deduplicates against existing `pending`/`failed` `move` proposals for the same event so repeated runs don't pile up duplicate proposals. On-demand only — no cron wiring yet. Verified end-to-end against the real burner calendar with seeded conflicts: flexible-vs-fixed (flexible moves, fixed untouched), flexible-vs-flexible at different priorities (lower priority moves), and re-running confirmed as a no-op (both skipped as already-pending) — this closes out Phase 2.

Also done alongside this: `GET /api/calendar/events` gained `from`/`to`/`maxResults`/`pageToken`/`q` query params (previously fixed at "upcoming, 50 max, no search") — response shape changed from a bare array to `{ events, nextPageToken }` to support pagination. See `backend-api-reference.md`.

## Phase 3.5 — NL-readiness gaps (found via pre-deployment audit, 2026-07-23)

Surfaced by walking 5 representative NL phrases through the current API surface before wiring up Phase 5's chat layer, to check nothing was missing — each item below is what blocked at least one phrase from mapping cleanly onto an existing endpoint. Small, targeted fixes; not a new feature area, and doesn't touch anything already built.

19. ✅ **Done.** **Direct task creation** (`POST /api/tasks`) — the only other way to insert into `tasks` is the Todoist-intake shape of `proposed_changes` (`change_type:'create'`, `category:'task'`, no `proposed_start`/`end`), built specifically to review *external, unreviewed* data before trusting it. A task the user states directly to the assistant ("I need a haircut by Monday," "new assignment due August 20th") isn't external data needing that same review, so it now has its own direct-insert path — same reasoning `habits` already uses (a task declaration isn't a calendar write, so it doesn't need the review queue). `title` required; `description`/`deadline`/`priority`/`duration_minutes`/`tags` all optional and left unset (not defaulted) at insert time, resolved at use-time elsewhere same as Todoist-sourced tasks; `source_system: 'manual'` — a value the schema's `tasks.source_system` check already enumerated but nothing had written yet.
20. **Source-system-aware auto-apply** — `AUTO_APPLY_CATEGORIES` only keys on `category`, not `source_system`, so there's no way to auto-apply chat-stated task creates while still holding Todoist imports back for review (the reason `task` is deliberately kept out of that whitelist today). Without this, every single thing the NL layer creates sits in the pending queue needing a second manual tap, even though the user just said it directly. Needs a way to combine `category` + `source_system` (or a new `source_system`/marker distinguishing "user said this in chat" from other `'manual'` writes) into the auto-apply decision.
21. ✅ **Done.** **Habit cadence: daily / every-N-days** — `habits.cadence` previously only supported `'weekly'`/`'monthly'`. Two additions: `'daily'` (a trivial extension — a fixed one-day calendar period, same model as weekly/monthly, via `lib/periodRanges.ts`'s new `getCurrentDayRange`) and `'interval'` (the real fix for "every other day" — full rolling-anchor model chosen of the options presented, over an approximate weekly+min-gap version or deferring it entirely). `'interval'` cadence stores a new `interval_days` column and is placed via genuinely different logic in `lib/habitPlacement.ts`'s `computeIntervalHabitContext`: it anchors its window to the habit's own actual last occurrence (real event or pending/failed proposal, whichever is more recent) rather than a fixed calendar boundary, so the next occurrence is never proposed less than `interval_days` after it — this is what a period-based approximation couldn't guarantee, since scoping was investigated and confirmed to have no cross-period spacing guarantee at all, even for the pre-existing `'weekly'`/`'monthly'` cadences (undocumented until now, out of scope to fix here). `habits_interval_requires_days` DB constraint enforces `interval_days` is set whenever `cadence: 'interval'`. Also fixed: `occurrence_duration_minutes` is now nullable with a `HABIT_DEFAULT_OCCURRENCE_DURATION_MINUTES` fallback (defaults to 30), resolved lazily via `lib/habits.ts`'s `resolveHabitOccurrenceDurationMinutes` — same pattern as `tasks.duration_minutes`/`TASK_DEFAULT_DURATION_MINUTES`.
22. ✅ **Done.** **A leisure/personal-time category** — `BurnerEventType` gained `'personal'` (`lib/eventMetadata.ts`), for freestanding blocks like "budget 2 hours of video games" that don't fit any existing category (`buffer` is semantically tied to a trigger event, every other category implies a specific engine). Needed no special-case handling anywhere — confirmed while scoping that `'task'` is the only category with any bespoke validation branches in `lib/proposedChanges.ts`, so `'personal'` slots in exactly like `meeting`/`fixed`: created via a plain `POST /api/proposed-changes` call, no dedicated engine, `flexible`/`priority` caller-supplied or defaulted the same generic way. New `#43A047` (green) `CATEGORY_COLORS` entry, distinct from all six existing category colors.
23. ✅ **Done.** **On-demand day-rebalancing** ("reduce today's workload") — `POST /api/calendar/reschedule` only resolves actual overlapping conflicts; there's no primitive for "nothing's conflicting, I just want less on my plate today." Narrower than the full **AI Planner** (item 11, above) — this is specifically "given today's schedule, propose moving/deferring some lower-priority flexible items elsewhere," not the whole "coherent daily view" orchestration. Implemented via `lib/dayRebalance.ts`'s `rebalanceWorkload` + `POST /api/calendar/rebalance`, reusing `lib/autoReschedule.ts`'s event-fetching and movability rule (exported `fetchSchedulableEvents` for this purpose) rather than duplicating it. Design chosen (of options presented): caller supplies an explicit `maxBusyMinutes` target ceiling (not a count, not an auto-detected working-hours threshold) — matches this backend's consistent explicit-params style; relocation search spans a caller-controlled `[to, searchTo)` window. One implementation detail worth flagging: the search floor is pinned to `to` regardless of the outer `searchTo` bound the user chose, because a relocation landing back inside `[from, to)` wouldn't reduce that window's total at all — only out-of-window moves count toward the target, so `to` is the only sensible floor.
    - **Three real bugs caught by live verification against the real calendar, all fixed before shipping:** (1) priority sort was inverted (moved the *most* important flexible item first instead of least — a copy-paste-style rank comparison error, confirmed live when a medium-priority item moved ahead of a low-priority one); (2) no claimed-interval tracking across candidates within one run, so two candidates could each independently pick the same still-genuinely-free slot and get proposed into an overlapping time (fixed by mirroring `lib/habitPlacement.ts`'s `claimedIntervals` pattern); (3) all-day events had their full multi-day span summed as literal "busy minutes" via the reused `fetchSchedulableEvents` (inherited from `lib/autoReschedule.ts`, where it doesn't matter for pairwise overlap-checking but badly distorts a sum) — fixed by exposing `isAllDay` on `SchedulableEvent` and excluding all-day events from this endpoint's accounting entirely. None of these three were caught by `tsc`/the unit test suite — all surfaced only by exercising the endpoint against real seeded calendar events.
24. ✅ **Done.** **Conflict-aware placement ("bump if movable")** — placing something at a specific requested time (`POST /api/tasks/{id}/schedule` with explicit `proposed_start`/`end`, or any `create`/`move` proposal) used to hit a hard wall the moment something's already there. Implemented as a new opt-in `bump_if_movable` field on `proposed_changes` (`POST /api/proposed-changes`, and threaded through `POST /api/tasks/{id}/schedule` as `bumpIfMovable`) — deliberately opt-in, not the new default, so existing "conflict = fail" callers are unaffected. `lib/proposedChanges.ts`'s `applyProposedChange` gained `attemptBump`: all-or-nothing (every conflicting event must be flexible and *strictly* lower priority, or nothing bumps — a partial bump wouldn't free the slot anyway), reusing the same flexible/priority movability rule as `lib/autoReschedule.ts`. Design confirmed with the user: the bump is a "hard manual option" — the occupant's relocation is created as its own ordinary `move` proposal but deliberately bypasses `AUTO_APPLY_CATEGORIES` (a new internal `skipAutoApply` option on `createProposedChange`, not exposed via any route) so it can never silently cascade onto the real calendar; the original request is left `failed` with a message naming the blocking proposal(s), reusing the existing failed-proposal retry flow (`POST /api/proposed-changes/{id}/approve`) rather than inventing new state — approve the relocation(s) first, then retry the original.
25. ✅ **Done.** **Recurring events** — surfaced by tracing "I have a new meeting every Thursday at 7pm." Built as Option B (synthesizing N individual `create` proposals, over native Google `recurrence`/RRULE) — chosen specifically because items 13/14 (event tags + bulk actions, built first as deliberate prep) solve Option B's usual weakness of needing a bespoke way to treat the whole series as one unit: every occurrence shares a fresh `series:<uuid>` tag, so "edit/cancel the whole series" is just `POST /api/calendar/bulk-edit` with that tag. Implemented via `lib/recurringEvents.ts`'s `planRecurringSeries` + `lib/recurringOccurrences.ts`'s `generateWeeklyOccurrences` (kept side-effect-free/separately unit-tested, mirroring `habitSpacing.ts`/`habitPlacement.ts`'s split) + `POST /api/calendar/recurring`. Weekly cadence with a configurable `interval_weeks` (supports "every other Tuesday"); DST-safe via Luxon zone-aware date math (unit-tested across both spring-forward and fall-back transitions) so a 7pm commitment stays 7pm local. Each occurrence is an ordinary `create` proposal, so it gets the exact same per-instance `detectConflicts` check at apply time as everything else — no new batch conflict-checking logic needed, which was the other half of Option B's appeal. **Confirmed with the user, resolving the exact question these notes originally flagged as unsolved:** `count` or `until` is required, no default horizon — no "silently ran out" failure mode, at the cost of no "recurring forever" in one call (an indefinite series just means calling this again later). Hard `MAX_OCCURRENCES = 260` (5 years weekly) sanity cap regardless, surfaced via a `truncated` response field.
