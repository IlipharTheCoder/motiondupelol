# Backend Build Order

A sequence for what to hand Claude (VS Code) next, phase by phase. Ordering logic, in short: **prove the connection ‚Üí build the one shared primitive everything else reuses ‚Üí put the safety pattern (human-in-the-loop) in place before anything auto-writes ‚Üí layer features on top ‚Üí chat/AI layer last, since it's a thin wrapper around everything below it.**

Each item below references its number from `ai-calendar-manager-spec.md`'s section 12 table, so you can cross-check the algorithmic/API/AI classification while building.

## Phase 1 ‚Äî Prove the calendar connection

The immediate next step ‚Äî everything after this depends on it actually working.

1. ‚úÖ **Done.** **`GET /api/calendar/events`** ‚Äî list events from the burner calendar using the service account credentials. No scheduling logic yet, just proves the auth chain works end to end (table #19). Implemented via `lib/googleCalendar.ts` (shared JWT client, full `calendar` scope) + `app/api/calendar/events/route.ts`; verified locally against the real burner calendar.
2. ‚úÖ **Done.** **External ‚Üí burner sync engine** ‚Äî using `syncToken` for incremental updates and `extendedProperties` source-ID tagging for dedup, per the design already locked into `architecture-plan.md` section 2a (table #19, #20). Implemented via `lib/calendarSync.ts` + `lib/eventMetadata.ts` + `POST /api/calendar/sync`, backed by two new Supabase tables (`calendar_sync_state`, `synced_events`) for resumable, efficient dedup. Source calendars are configured via `GOOGLE_SOURCE_CALENDAR_IDS` (`calendarList.list()` doesn't work for service accounts ‚Äî see `architecture-plan.md` section 2a), each tagged with a `sourceLabel`; first backfill per calendar capped 1 year out; Google writes retry with backoff to survive the Calendar API's write burst quota. Verified end-to-end against 5 real calendars ‚Äî full backfill, idempotent re-run (steady-state incremental, zero changes), and correct `sourceLabel` tagging all confirmed.
   - **Bug found & fixed (2026-07-22):** a batched end-of-page `synced_events` flush meant a mid-page function kill could lose already-successful Google writes, causing the next sync to recreate the same events as duplicates ‚Äî 121 orphaned duplicates were produced this way in one source calendar. Fixed by flushing each event's `synced_events` row immediately after its own Google write succeeds. A maintenance endpoint, `POST /api/calendar/sync/dedupe`, was added to clean up existing duplicates (and any future ones) ‚Äî see `backend-api-reference.md`. Verified: cleanup ran to zero duplicates, and a subsequent sync produced `created: 0` across all calendars.
   - **Default priority changed to `critical` (2026-07-22):** synced events defaulted to `medium`; changed to `critical` per the user ‚Äî synced events are always `flexible: 'false'` (hard to move) and shouldn't need to move anyway. Only affects events synced from here on; already-synced events keep whatever priority they were written with until they're next touched by a real upstream change (the sync engine skips unchanged events via etag, so this isn't retroactive).

## Phase 2 ‚Äî The shared primitive + the safety pattern

Build these before any feature that writes to the calendar ‚Äî everything in Phase 3 is "apply this primitive with different constraints," and nothing should auto-write without the review queue existing first.

3. ‚úÖ **Done.** **Free-slot finding / conflict detection** (table #1, #2) ‚Äî the one piece of logic that tasks, habits, focus time, buffer insertion, and the booking page will all reuse. Worth over-investing in correctness here specifically. Implemented as a clean pure/IO split: `lib/intervals.ts` (pure interval math ‚Äî merge/subtract/pad/filter, unit-tested), `lib/workingHours.ts` (DST-correct working-window generation via `luxon`, unit-tested including real spring-forward/fall-back transitions), `lib/busyIntervals.ts` (Google fetch + per-event-timezone normalization, unit-tested), `lib/freeSlots.ts` (`findFreeSlots`/`detectConflicts` orchestration), `lib/schedulingConfig.ts` (`HOME_TIMEZONE`/`WORKING_HOURS_START`/`WORKING_HOURS_END`/`WORKING_DAYS` env vars, defaults `America/New_York` 10:00‚Äì18:00 Mon‚ÄìFri). Exposed via `GET /api/calendar/free-slots` and `GET /api/calendar/conflicts` for verification against the real burner calendar. 47 unit tests (`npm test`), every non-cancelled burner event counts as busy regardless of `flexible` tag ‚Äî deciding what to do about conflicts is explicitly deferred to item 5.
4. ‚úÖ **Done.** **Proposed-changes / review queue** ‚Äî a data model (separate from calendar events) plus approve/reject endpoints. This is the human-in-the-loop mechanism from the spec's AI Assistant section ‚Äî every scheduling feature from here on proposes into this queue rather than writing directly. Implemented via a new `proposed_changes` Supabase table + `lib/proposedChanges.ts` + `GET/POST /api/proposed-changes` + `POST /api/proposed-changes/{id}/approve|reject`. Per-`change_type` field validation (`create`/`move`/`update`/`delete`); a 4-state status machine (`pending ‚Üí applied | rejected | failed`, no separate "approved" state since applying happens synchronously); an `AUTO_APPLY_CATEGORIES` whitelist env var (empty/unset = nothing auto-applies, per your explicit choice to build this now rather than defer it) that applies a proposal immediately at creation time instead of waiting for approval; a conflict recheck via `detectConflicts` (item 3's primitive) on every `create`/`move` apply, failing safe with a descriptive error rather than double-booking. Verified end-to-end against the real burner calendar: manual approve, auto-apply (both success and a real-conflict failure), reject, retry-from-failed, and 409s on already-decided rows.
   - **Data-model alignment + additions (2026-07-22):** clarified the real model with the user ‚Äî a `tasks` database (not yet built, Phase 3) and calendar events are separate; a block *can* optionally tie back to a task but doesn't have to; synced external events are always non-flexible (already true). Changed `priority` from a numeric `1-5` scale to `critical`/`high`/`medium`/`low` (matches how the user actually thinks about events ‚Äî doctors' appointments/classes are `critical`); `colorTag` is now always derived from `category` (`lib/eventMetadata.ts`'s `CATEGORY_COLORS`) rather than freely settable, so it's never blank; added a `deadline` field (a "must be done by" constraint, independent of where a block is actually scheduled ‚Äî no logic reads it yet, that's Phase 3 item 7); `GET /api/calendar/events` now surfaces `category`/`priority`/`deadline`/`colorTag`/`origin` per event; every create/approve/reject response now includes a plain-language `message` alongside `status`. "All-day note vs. block" isn't a separately stored type ‚Äî it's inferred from Google's own all-day-vs-timed event shape, same as `lib/busyIntervals.ts` already does for scheduling.
5. ‚úÖ **Done.** **Auto-reschedule on conflict** (table #3) ‚Äî mechanical once #3 and #4 exist: detect overlap, propose a move, wait for approval. Implemented via `lib/autoReschedule.ts` (`findAndProposeReschedules`) + `POST /api/calendar/reschedule`. Pairwise conflict scan over a time window using `intervalsOverlap` (item 3); movability comes from an event's own `flexible` metadata, not `category` ‚Äî if both sides of a conflict are flexible, the lower-priority one moves (`critical` > `high` > `medium` > `low`, the priority scale from item 4's rework), tie-broken deterministically by `eventId`; unresolvable (both non-flexible) conflicts are reported, not forced. Replacement slot is the first opening `findFreeSlots` returns of at least the event's own duration; proposes via the normal `createProposedChange` (item 4) so `AUTO_APPLY_CATEGORIES` still applies immediately where whitelisted. Deduplicates against existing `pending`/`failed` `move` proposals for the same event so repeated runs don't pile up duplicate proposals. On-demand only ‚Äî no cron wiring yet. Verified end-to-end against the real burner calendar with seeded conflicts: flexible-vs-fixed (flexible moves, fixed untouched), flexible-vs-flexible at different priorities (lower priority moves), and re-running confirmed as a no-op (both skipped as already-pending) ‚Äî this closes out Phase 2.

Also done alongside this: `GET /api/calendar/events` gained `from`/`to`/`maxResults`/`pageToken`/`q` query params (previously fixed at "upcoming, 50 max, no search") ‚Äî response shape changed from a bare array to `{ events, nextPageToken }` to support pagination. See `backend-api-reference.md`.

## Phase 3.5 ‚Äî NL-readiness gaps (found via pre-deployment audit, 2026-07-23)

Surfaced by walking 5 representative NL phrases through the current API surface before wiring up Phase 5's chat layer, to check nothing was missing ‚Äî each item below is what blocked at least one phrase from mapping cleanly onto an existing endpoint. Small, targeted fixes; not a new feature area, and doesn't touch anything already built.

19. ‚úÖ **Done.** **Direct task creation** (`POST /api/tasks`) ‚Äî the only other way to insert into `tasks` is the Todoist-intake shape of `proposed_changes` (`change_type:'create'`, `category:'task'`, no `proposed_start`/`end`), built specifically to review *external, unreviewed* data before trusting it. A task the user states directly to the assistant ("I need a haircut by Monday," "new assignment due August 20th") isn't external data needing that same review, so it now has its own direct-insert path ‚Äî same reasoning `habits` already uses (a task declaration isn't a calendar write, so it doesn't need the review queue). `title` required; `description`/`deadline`/`priority`/`duration_minutes`/`tags` all optional and left unset (not defaulted) at insert time, resolved at use-time elsewhere same as Todoist-sourced tasks; `source_system: 'manual'` ‚Äî a value the schema's `tasks.source_system` check already enumerated but nothing had written yet.
20. **Source-system-aware auto-apply** ‚Äî `AUTO_APPLY_CATEGORIES` only keys on `category`, not `source_system`, so there's no way to auto-apply chat-stated task creates while still holding Todoist imports back for review (the reason `task` is deliberately kept out of that whitelist today). Without this, every single thing the NL layer creates sits in the pending queue needing a second manual tap, even though the user just said it directly. Needs a way to combine `category` + `source_system` (or a new `source_system`/marker distinguishing "user said this in chat" from other `'manual'` writes) into the auto-apply decision.
21. ‚úÖ **Done.** **Habit cadence: daily / every-N-days** ‚Äî `habits.cadence` previously only supported `'weekly'`/`'monthly'`. Two additions: `'daily'` (a trivial extension ‚Äî a fixed one-day calendar period, same model as weekly/monthly, via `lib/periodRanges.ts`'s new `getCurrentDayRange`) and `'interval'` (the real fix for "every other day" ‚Äî full rolling-anchor model chosen of the options presented, over an approximate weekly+min-gap version or deferring it entirely). `'interval'` cadence stores a new `interval_days` column and is placed via genuinely different logic in `lib/habitPlacement.ts`'s `computeIntervalHabitContext`: it anchors its window to the habit's own actual last occurrence (real event or pending/failed proposal, whichever is more recent) rather than a fixed calendar boundary, so the next occurrence is never proposed less than `interval_days` after it ‚Äî this is what a period-based approximation couldn't guarantee, since scoping was investigated and confirmed to have no cross-period spacing guarantee at all, even for the pre-existing `'weekly'`/`'monthly'` cadences (undocumented until now, out of scope to fix here). `habits_interval_requires_days` DB constraint enforces `interval_days` is set whenever `cadence: 'interval'`. Also fixed: `occurrence_duration_minutes` is now nullable with a `HABIT_DEFAULT_OCCURRENCE_DURATION_MINUTES` fallback (defaults to 30), resolved lazily via `lib/habits.ts`'s `resolveHabitOccurrenceDurationMinutes` ‚Äî same pattern as `tasks.duration_minutes`/`TASK_DEFAULT_DURATION_MINUTES`.
22. ‚úÖ **Done.** **A leisure/personal-time category** ‚Äî `BurnerEventType` gained `'personal'` (`lib/eventMetadata.ts`), for freestanding blocks like "budget 2 hours of video games" that don't fit any existing category (`buffer` is semantically tied to a trigger event, every other category implies a specific engine). Needed no special-case handling anywhere ‚Äî confirmed while scoping that `'task'` is the only category with any bespoke validation branches in `lib/proposedChanges.ts`, so `'personal'` slots in exactly like `meeting`/`fixed`: created via a plain `POST /api/proposed-changes` call, no dedicated engine, `flexible`/`priority` caller-supplied or defaulted the same generic way. New `#43A047` (green) `CATEGORY_COLORS` entry, distinct from all six existing category colors.
23. ‚úÖ **Done.** **On-demand day-rebalancing** ("reduce today's workload") ‚Äî `POST /api/calendar/reschedule` only resolves actual overlapping conflicts; there's no primitive for "nothing's conflicting, I just want less on my plate today." Narrower than the full **AI Planner** (item 11, above) ‚Äî this is specifically "given today's schedule, propose moving/deferring some lower-priority flexible items elsewhere," not the whole "coherent daily view" orchestration. Implemented via `lib/dayRebalance.ts`'s `rebalanceWorkload` + `POST /api/calendar/rebalance`, reusing `lib/autoReschedule.ts`'s event-fetching and movability rule (exported `fetchSchedulableEvents` for this purpose) rather than duplicating it. Design chosen (of options presented): caller supplies an explicit `maxBusyMinutes` target ceiling (not a count, not an auto-detected working-hours threshold) ‚Äî matches this backend's consistent explicit-params style; relocation search spans a caller-controlled `[to, searchTo)` window. One implementation detail worth flagging: the search floor is pinned to `to` regardless of the outer `searchTo` bound the user chose, because a relocation landing back inside `[from, to)` wouldn't reduce that window's total at all ‚Äî only out-of-window moves count toward the target, so `to` is the only sensible floor.
    - **Three real bugs caught by live verification against the real calendar, all fixed before shipping:** (1) priority sort was inverted (moved the *most* important flexible item first instead of least ‚Äî a copy-paste-style rank comparison error, confirmed live when a medium-priority item moved ahead of a low-priority one); (2) no claimed-interval tracking across candidates within one run, so two candidates could each independently pick the same still-genuinely-free slot and get proposed into an overlapping time (fixed by mirroring `lib/habitPlacement.ts`'s `claimedIntervals` pattern); (3) all-day events had their full multi-day span summed as literal "busy minutes" via the reused `fetchSchedulableEvents` (inherited from `lib/autoReschedule.ts`, where it doesn't matter for pairwise overlap-checking but badly distorts a sum) ‚Äî fixed by exposing `isAllDay` on `SchedulableEvent` and excluding all-day events from this endpoint's accounting entirely. None of these three were caught by `tsc`/the unit test suite ‚Äî all surfaced only by exercising the endpoint against real seeded calendar events.
24. ‚úÖ **Done.** **Conflict-aware placement ("bump if movable")** ‚Äî placing something at a specific requested time (`POST /api/tasks/{id}/schedule` with explicit `proposed_start`/`end`, or any `create`/`move` proposal) used to hit a hard wall the moment something's already there. Implemented as a new opt-in `bump_if_movable` field on `proposed_changes` (`POST /api/proposed-changes`, and threaded through `POST /api/tasks/{id}/schedule` as `bumpIfMovable`) ‚Äî deliberately opt-in, not the new default, so existing "conflict = fail" callers are unaffected. `lib/proposedChanges.ts`'s `applyProposedChange` gained `attemptBump`: all-or-nothing (every conflicting event must be flexible and *strictly* lower priority, or nothing bumps ‚Äî a partial bump wouldn't free the slot anyway), reusing the same flexible/priority movability rule as `lib/autoReschedule.ts`. Design confirmed with the user: the bump is a "hard manual option" ‚Äî the occupant's relocation is created as its own ordinary `move` proposal but deliberately bypasses `AUTO_APPLY_CATEGORIES` (a new internal `skipAutoApply` option on `createProposedChange`, not exposed via any route) so it can never silently cascade onto the real calendar; the original request is left `failed` with a message naming the blocking proposal(s), reusing the existing failed-proposal retry flow (`POST /api/proposed-changes/{id}/approve`) rather than inventing new state ‚Äî approve the relocation(s) first, then retry the original.

**Both items 17/18 built together** as one `POST /api/chat` route ‚Äî a thin conversational wrapper around every backend capability built in Phases 1-4 and 3.5, exactly the "cheap to add once the underlying actions exist" framing above. Implemented as:
- A **manual agentic tool-calling loop** (`lib/nlLoop.ts`), not the Anthropic SDK's beta Tool Runner ‚Äî this repo's own convention (`CLAUDE.md`: "prefer plain algorithmic code... wherever possible") plus two behaviors a generic runner's iteration protocol doesn't fit cleanly: a hard 6-iteration cap, and `ask_clarifying_question` needing to short-circuit the whole loop with a distinct response shape rather than feed a tool_result back in.
- **31 tools** (`lib/nlToolManifest.ts`, Read/Propose/Decision/Direct-write/Meta) mapped 1:1 onto real `lib/` functions (`lib/nlToolDispatch.ts`) ‚Äî every calendar-affecting tool still creates a `proposed_changes` row; nothing bypasses the review queue. Grew from the spec's original ~24 estimate once reconciled against the actual, fuller Phase 1-4 API surface (e.g. separate `plan_focus_time`/`plan_buffer_time`/`plan_tasks`/`plan_habits`/`rebalance_day` tools rather than collapsing distinct planners into one) ‚Äî each one maps onto a real, already-built, already-tested capability rather than an artificial cut to hit a round number.
- **Model: `claude-haiku-4-5`**, not the `claude-api` skill's own stated Opus-4.8-unless-told-otherwise default ‚Äî the user explicitly overrode this ("we should be using haiku for basically everything, sometimes sonnet if a task is heavy. opus + fable should NEVER BE USED"), which is exactly the condition that default defers to. `NL_MODEL` is a single named constant (`lib/nlLoop.ts`) so an escalation to Sonnet for a specific heavy step stays a deliberate, visible one-line change.
- **A prerequisite refactor** (done first, zero behavior change, live-verified via before/after `curl` comparisons): four "declaration" tables (`tasks`, `habits`, `scheduling_rules`, `capability_requests`) had their POST/PATCH validation written inline in the route handler by deliberate existing convention ‚Äî extracted into `lib/tasksWrite.ts`/`tasksQuery.ts`, `lib/habitsWrite.ts`/`habitsQuery.ts`, `lib/schedulingRulesQuery.ts` (extended), `lib/capabilityRequestsWrite.ts`, plus a new `lib/calendarEvents.ts` (no prior lib wrapper existed for the human-readable calendar-listing shape `get_calendar_events` needed). Each route is now a thin `isAuthorized ‚Üí parse ‚Üí call lib fn ‚Üí map error to status` wrapper, reusing `ValidationError`/`NotFoundError` from `lib/proposedChanges.ts`.
- **Prompt-caching design bug caught and fixed before building**, not found live: the source spec's own framing ("put Part 2 context in a cacheable prefix") would have put the resolved-time-anchors block ‚Äî which recomputes `now` on literally every request ‚Äî before or inside the cache breakpoint, defeating caching entirely (a strict prefix match; any byte change invalidates everything after it). Split instead into a **stable/cacheable** system block (`lib/nlToolManifest.ts`'s tool schemas + `lib/nlSystemPrompt.ts`'s behavioral rules/enums, one `cache_control` breakpoint on the end of it) and a **volatile** block appended after the breakpoint (resolved time anchors, calendar digest, open state, scheduling config+rules ‚Äî `lib/nlContext.ts`/`lib/nlContextQuery.ts`, re-fetched fresh every call since active rules can change mid-session via a live `PATCH`).
- **Conversation persistence**: lightweight, text-only (`chat_conversations`/`chat_messages` ‚Äî `backend-schema.md`) ‚Äî only final user/assistant text is stored, never `tool_use`/`tool_result` blocks; "open state" (pending proposals/groups, recent decisions) is recomputed fresh from the live tables every turn rather than replayed from history. Confirmed via `AskUserQuestion` against three alternatives before building.
- **Confirmed v1 posture**: nothing auto-applies from chat (item 20, source-system-aware auto-apply, stays explicitly out of scope) ‚Äî chosen via `AskUserQuestion` over building item 20 first. Every chat-originated write lands `pending`, same manual-tap requirement as any other `source_system`.

**Live-verified end-to-end (2026-07-24)**, against the real burner calendar/Supabase, not just `tsc`/unit tests (this project's own repeated finding: live testing catches real bugs unit tests miss):
- A calendar-digest query ("what's on my calendar this week") ‚Äî correct real events, correctly formatted.
- A scheduling rule created via chat (`create_scheduling_rule`, direct-write, no proposal), confirmed to exist via the raw API, then a second-turn attempt to book a rule-violating time ‚Äî the write-gate correctly rejected it at *approval* time with the exact rule-violation message (`checkCandidateAgainstRules`), proving the gate applies through the NL path exactly as it does through the raw API. Cleaned up after (rule set inactive, then deleted).
- A `propose_batch` call ‚Äî two proposals created under one `group_id`, correctly round-tripped in the response; `approve_proposal_group` applied both to the real calendar.
- A two-turn **"undo that"** sequence ‚Äî confirmed the model correctly chains `revert_proposal` ‚Üí `approve_proposal` within one loop (both the pending revert and its subsequent `applied` state showed up in the accumulated `proposals` response) rather than stopping after `revert_proposal` and prematurely claiming success.
- `ask_clarifying_question` plumbing ‚Äî confirmed the loop correctly short-circuits with the distinct `clarification` response field when the tool is actually called. **Real finding, not a bug:** the model doesn't always choose to call this tool for an ambiguous request ‚Äî it sometimes just replies with a clarifying question as plain text instead (functionally identical to the user; the response's `reply` field always carries the question either way, `clarification` is a best-effort structured signal on top, not guaranteed).
- `log_capability_gap` ‚Äî confirmed a real row lands in `capability_requests`.
- **A genuine model-behavior finding from live testing, contained by the architecture it was designed to be contained by:** asked the model to "delete ZZTEST batch B" without giving an explicit event id ‚Äî its first attempt within the loop targeted the wrong event (a real, unrelated "Mock Teach-In" meeting), before self-correcting and deleting the right one later in the same turn. The wrong delete proposal was only ever created `pending` ‚Äî the model never called `approve_proposal` on it, so nothing was actually touched on the real calendar. This is exactly the review-queue's job; caught and rejected during verification, never a live risk.
- **Prompt caching** ‚Äî confirmed real, working, and worth the design effort: the actual stable system content (`NL_TOOLS` + `NL_BEHAVIORAL_RULES`, ~6,700 tokens) showed `cache_creation_input_tokens` on a cold write and `cache_read_input_tokens` matching it exactly on a subsequent call. Switched the breakpoint's TTL from the API default (5 minutes) to `1h` ‚Äî this is a low-frequency single-user app where separate chat sessions during the day are more likely to be >5 minutes apart than not, so the longer window meaningfully raises the real-world hit rate for a negligible extra cache-write cost at this token volume.

All test data (calendar events, scheduling rule, proposals, capability request) cleaned up after.
